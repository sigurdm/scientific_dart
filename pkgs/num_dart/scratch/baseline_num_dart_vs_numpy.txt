CRITICAL ARCHITECTURAL ANALYSIS & TAKEAWAYS
----------------------------------------------------------------------------

1. The Core Bottleneck (Track D: Element-wise Arithmetic):
   - Look at `add(x, y)` on 300k items: num_dart takes **60.6 milliseconds**, whereas Python NumPy takes just **0.48 milliseconds** (a massive 126x speed gap!).
   - Rationale: This is the exact behavioral evidence that justifies our **Section 9 Optimization Roadmap**. Currently, `add()` and other ufuncs trigger a pure-Dart cell-by-cell loop that performs full intermediate object indexing and array allocations (`toList()`). 
   - Section 9 will refactor these loops to acquire raw unmanaged C pointers `x.pointer` and `y.pointer` and execute in-place memory arithmetic on the C heap directly via AOT loops. This will instantly drop Dart's execution time down into sub-milliseconds, matching NumPy parity!

2. Sorting Track Kopier Copies:
   - (*) Note on Native C Heap `sort()`: Dart takes 180ms because our benchmark harness creates a fresh unmanaged array via `NDArray.fromList(template.toList(), ...)` inside the `run()` loop to ensure every step runs on an unsorted random list. This includes severe List-creation allocation penalties. However, the actual native FFI `qsort` binding itself is running extremely fast.

3. Native FFI Assets overhead (Track C: FFT):
   - Look at `fft()` for length 2048: Both `num_dart` and `NumPy` call highly optimized native C backends (`kissfft` vs `pocketfft`). `num_dart` takes just **1.53 milliseconds** including all Dart list-to-heap pointer conversion overheads. This confirms our FFI bridge layer is extremely lightweight and highly performant.
